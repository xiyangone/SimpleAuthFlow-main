# SimpleAuthFlow

一个围绕自动化注册和授权流程的 Chrome 扩展。

## 项目简介

SimpleAuthFlow 是一个用于简化 ChatGPT OAuth 注册和授权流程的浏览器自动化扩展，目标是尽可能降低使用门槛，让整套流程更直接、更省事。

这个版本最核心的特点只有一个：

**开箱即用，无需任何额外配置。**

不需要你手动准备自己的邮箱，不需要额外填写手机号，也不需要再改一堆插件配置。  
装好之后就可以直接开始，尽量把使用门槛降到最低。

## 使用前提

本项目的使用前提只有两个：

1. 装好 CPA
2. 装好本插件

完成后即可开始使用，不需要你额外填写自己的邮箱和手机号。

## 安装和使用方式

1. 打开 `chrome://extensions/`
2. 开启“开发者模式”
3. 点击“加载已解压的扩展程序”
4. 选择本项目文件夹
5. 点击浏览器工具栏中的扩展图标，打开侧边面板
6. VPS 一栏可直接留空，默认会使用 `http://127.0.0.1:8317/management.html#/oauth`；如果你的 CPA 地址不同，再手动填入对应鉴权地址
7. 点击“自动”开始运行
8. 如需人工接管验证码，切换到“手动模式”后，步骤 4/7 会暂停并弹出验证码输入框，提交后自动继续

## 密码说明

侧边栏里的“密码”是用于 OpenAI 注册/登录流程的账号密码，不是系统密码或 VPS 密码。

- 留空：插件会自动生成随机密码
- 手动填写：插件使用你填写的自定义密码
- 建议：使用独立临时密码，不要复用你常用主密码

## 近期更新

基于最近这一轮调整，当前版本额外补上了下面几项：

### 0. 新增手动模式核心闭环（2026/4/9）

- 新增“自动模式 / 手动模式”切换
- 手动模式下，步骤 4/7 不再自动拉取验证码，而是等待你手动输入
- 侧边栏新增验证码弹层，输入 6 位验证码后自动续跑
- 自动模式保持不变，仍沿用 Burner Mailbox 自动取码流程

### 1. 侧边栏与运行时提示已统一汉化

- 侧边栏按钮、状态、步骤名称、占位符统一改为中文
- Toast、确认弹窗、日志区主要提示改为中文
- 运行时错误和关键流程日志也做了统一收口，排查时更直观

### 2. 注册验证码与登录验证码会更早触发重发

在步骤 4 和步骤 7 里，现在会在开始邮箱轮询前先主动触发一次“重新发送验证码”，而不是只在多轮扫描失败后才补发。  
这样做的目的，是尽量避免“第一次验证码根本没有发出来，结果前几轮扫描全都白等”的情况。

同时，原有“连续扫描失败后再次触发重发”的兜底逻辑仍然保留，因此现在是：

- 先预先触发一次重发
- 若后续仍未收到，再走原有补偿重发流程

### 3. 自动化标签页固定在同一个浏览器窗口

为了减少运行时窗口乱跳、标签页散到不同窗口的问题，现在新开的自动化标签页会尽量固定在同一个浏览器窗口中复用和创建。  
这样在以下场景里会更稳定：

- 打开 OAuth 页面
- 打开邮箱页面
- 返回 CPA 面板做最终确认

### 4. 增加了定向回归检查

当前仓库已经补了两类定向校验：

- 文案汉化回归检查
- 自动重发验证码 / 固定单窗口行为检查

这样后续继续改动时，至少可以更快发现这两类行为是否被回归破坏。

## 为什么做这个版本

在实际使用过程中发现，DuckDuckGo 相关邮箱方案最近可能使用人数较多，经常出现以下问题：

- 验证码很久才收到
- 验证码根本收不到
- 导致鉴权过期
- 导致自动化流程中断

因此，这个版本主要围绕“更直接、更省心的个人使用体验”进行调整，目标不是复杂配置，而是尽量减少额外准备和中断情况。

## 项目定位

本项目主打个人使用，能够满足个人日常的 Codex 使用需求。  
它并不是为大规模操作场景设计的工具，整体思路更偏向轻量、直接、够用。

在默认不额外修改配置的情况下，平均大约一分钟可以完成一个流程。  
如果完全不改任何东西，按当前设置，一天大约可以稳定处理 5 次左右。

目前限制主要来自邮箱获取方式：同一天内可能会将同一请求环境判定为同一台机器，因此默认每天限制约 5 次。  
如果某天确实需要更多次数，通常更换一个梯子节点即可继续使用。

从目前的现象看，目标站点大概率至少结合了请求头与 IP 地址进行判定，但这里只是基于实际使用过程的初步观察，没有进一步做更细的验证。

## 主要改动

本项目基于开源项目 [StepFlow-Duck](https://github.com/whwh1233/StepFlow-Duck) 的 `v1.0` 版本，以及“寸铁”所做的二次修改版本继续调整而来。  
当前这一版属于三次修改版本，在继承前面成果的基础上，继续围绕“个人使用”“开箱即用”“减少配置”做了进一步优化。

当前版本的主要改动包括：

### 1. 去掉原有个人配置依赖

删除了原插件中需要用户手动配置的邮箱、Duck 等相关内容，改为安装后即可直接使用，尽量减少前置准备。

### 2. 强化开箱即用体验

这个版本的重点就是：  
**不折腾配置，装好就跑。**

因为主要面向个人使用，可以直接本地运行 CPA 并本地配合使用，因此整体流程尽量围绕本地直接可用来设计，减少额外配置步骤和人工介入。

### 3. 简化本地 CPA 场景下的回调与验证过程

因为主要为个人使用，可以直接本地运行 CPA 并配合使用。  
因此将原有最后的回调与验证过程做了适配调整：

- 支持原有条件下的验证方式
- 也支持在出现授权完成页时继续处理
- 同时兼容本地上传和服务器回调两种方式

### 4. 提高邮箱发送流程容错

在注册过程中，偶尔会出现邮件没有正常发出的情况。  
现在会在检测到异常时自动返回重新触发发送，再继续等待验证码。

当前逻辑为：

- 自动重试最多 3 次
- 连续失败后才判定为失败

实际使用中，大多数情况下第一次即可成功，整体成功率相对稳定。

### 5. 增加 OpenAI 协议确认场景的自动化处理

实际使用时，OpenAI 有时候会突然跳一个“需要同意协议”或者“继续使用确认”的页面。  
这种情况以前很容易把流程卡住。

现在这一类情况也加了自动化处理，检测到相关页面后会自动继续，尽量避免因为额外弹窗导致整套流程中断。

### 6. 增加邮箱安全验证等待与继续机制

在使用 Burner Mailbox 的过程中，可能会在以下几个环节遇到网站的人机验证：

- 注册前自动获取邮箱
- 第一次进入邮箱查看注册验证码
- 第二次进入邮箱查看登录验证码

以前一旦遇到这一类验证页面，流程就很容易停住，用户还需要自己判断当前到底是哪里出了问题。  

现在针对这一类情况增加了专门的等待与继续处理：

- 检测到安全验证页面后，会明确提示用户先去邮箱页完成人机验证
- 用户完成验证后点击继续，流程会自动等待页面恢复正常
- 页面恢复后会直接继续后续获取邮箱或拉取验证码流程

这样可以尽量避免因为 Burner Mailbox 临时触发安全验证而导致整套自动化流程直接中断。

### 7. 默认 CPA 地址可直接留空使用

为了进一步降低本地使用门槛，现在侧边栏中的 VPS 一栏支持直接留空。  
如果不手动填写地址，默认会自动使用：

`http://127.0.0.1:8317/management.html#/oauth`

如果你的 CPA 并不是这个地址，再按实际情况手动改成你自己的即可。

### 8. 修复 Burner Mailbox 当前页面结构兼容问题

Burner Mailbox 的页面结构和按钮布局与之前相比有了一些变化。  
旧逻辑在部分情况下会出现下面这些问题：

- 已经回到邮箱页，但插件仍然误判为还没恢复
- 点击继续后继续卡在等待状态
- 当前页面结构下无法稳定识别 Copy / Refresh / New 这些操作按钮

现在这一部分已做了兼容处理：

- 兼容当前 Burner Mailbox 的新按钮布局
- 人机验证结束后能够更准确判断邮箱页是否真的恢复正常
- 减少“页面已经好了但流程还在等”的卡死情况

### 9. 修复验证码误取旧邮件的问题

以前在拉取验证码时，如果收件箱里已经有旧邮件，某些情况下可能会误拿到旧验证码，导致后续验证失败或重复重试。

现在这一段逻辑增加了更严格的筛选：

- 会校验当前邮箱页和目标邮箱是否一致
- 会优先按照本轮流程的时间窗口筛选验证码邮件
- 避免回退时误拿到更早的旧验证码

这样可以减少“明明拿到验证码了但还是不通过”的情况。

### 10. 修复生日填写在不同页面形态下的兼容问题

OpenAI 的注册页在不同环境下，生日区域并不总是同一种结构。  
有时是 React Aria 的 dropdown，下拉式选择；有时又可能是 spinbutton 或 hidden input 的组合。

当前版本针对这一块做了兼容补强：

- 保留 dropdown 生日选择逻辑
- 同时兼容 spinbutton 与 hidden birthday input 的回填方式
- 当页面里识别不到生日控件时，不会直接把整套流程卡死，而是按当前页面状态继续往下提交

这样可以尽量降低因为生日控件形态变化导致流程中断的概率。

### 11. 减少首次验证码迟迟不到导致的空等

参考社区里对多页自动化版本的实践，现在在关键验证码步骤里会更早触发重新发送。  
对“第一次没发出来，后面才收到”的场景更友好，也能减少前几轮无效扫描带来的时间浪费。

### 12. 缓解自动化过程中浏览器窗口乱跳

自动化流程里不同来源页面会尽量复用同一个浏览器窗口，避免标签页在多个窗口之间来回弹出。  
整体观察和手动接管时会更顺手，也更方便排查问题。

## 设计目标

SimpleAuthFlow 不追求复杂的能力堆叠，而是更偏向下面几个方向：

- 开箱即用
- 无需任何额外配置
- 少手动操作
- 少中断
- 尽量本地即可跑通
- 更适合个人长期自用
